## 2. 总体架构

### 2.1 设计原则

1. **分层解耦**：`cmd` → `internal` → `shortcuts`，禁止跨层乱调用  
2. **基建先行**：配置、鉴权、HTTP、输出先于业务命令  
3. **终局架构一次搭建**：目录按终局设计，后续只叠加能力与域  
4. **体验统一**：三级命令、全局参数、stdout/stderr 分离对标飞书  

### 2.2 分层架构图

```text
┌─────────────────────────────────────────────────────────────┐
│  用户 / Shell 脚本 / AI Agent（AGENTS.md + Skill）           │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  cmd/              Cobra 注册、全局 flag，无业务逻辑             │
│  config / auth / api / schema / service / skill / completion │
│  root.go → shortcuts/item                                    │
└───────────────────────────┬─────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│ shortcuts/    │   │ cmd/service   │   │ cmd/api       │
│ item          │   │ 读 meta_data  │   │ 原始 HTTP     │
│ list/+list    │   │ v1.2.0        │   │ GET/POST/…    │
│ save/get-…    │   │               │   │               │
└───────┬───────┘   └───────┬───────┘   └───────┬───────┘
        │                   │                   │
        └───────────────────┼───────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  shortcuts/common：runner、RunGET/RunPOST/RunPOSTForm…        │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  internal/cmdutil/Factory → config + auth + client           │
│  internal/client：重试、分页、Header accessToken、连接池/熔断   │
│  internal/output：{ok,data,error,hint}、csv/ndjson、彩色 table │
```

### 2.3 目录职责

| 目录 | 职责 |
|------|------|
| `cmd/` | 命令注册、全局参数 |
| `internal/config` | Viper、`config init` 模板（erp1 单环境） |
| `internal/auth` | 密钥链 `accessToken` |
| `internal/client` | HTTP、重试、`--page-all`、连接池/熔断 |
| `internal/registry` | `meta_data.json`（嵌入二进制） |
| `shortcuts/common` | runner、路径与各类 Run* |
| `shortcuts/item` | 商品 list/count/get-detail/save |
| `scripts/fetch_meta/` | 可选 OpenAPI 拉取元数据（`fetch_meta.sh`） |
| `skills/` | `SKILL.md`（`kuaimai-shared`、`kuaimai-item`） |
| `pkg/` | `sanitize`、`logger`、`convert`、`util`（无业务域逻辑） |
| `internal/audit` | 业务命令审计行写入 `~/.kuaimai-cli/audit.log` |

## 3. 三级命令体系

| 层级 | 飞书示例 | kuaimai-cli 示例 | 路径来源 |
|------|----------|------------------|----------|
| **1. 快捷 / 标准业务** | `calendar +agenda` | `item +list`、`item save` | `shortcuts/item/item.go` |
| **2. 元数据 API** | `calendar events list` | `service item list` | `meta_data.json` |
| **3. 原始兜底** | `api GET /open-apis/...` | `api POST /item/stock/queryList` | 终端传入 path |

### 3.1 两条路径不要混淆

```text
kuaimai-cli item +list --body '{"title":"关键字"}'
  → shortcuts/item/item.go
  → POST form /item/stock/queryList
  → 不读 meta_data 做路由（路径应与 meta 一致）

kuaimai-cli service item list --body '{}'
  → cmd/servicecmd + registry
  → meta_data.json 中 item.list → POST /item/stock/queryList
```

改接口时建议 **同时更新** `shortcuts/item/item.go` 与 `meta_data.json`。

### 3.2 当前 meta_data.json（v1.2.0）

共 **1 个 service**、**4 个 operation**：

| operation | method | path | 说明 |
|-----------|--------|------|------|
| list | POST form | `/item/stock/queryList` | 库存列表（含 `title` 筛选） |
| count | POST form | `/item/stock/queryCount` | 总数 |
| get-detail | GET | `/item/getItemDetail` | 详情 |
| save | POST JSON | `/item/saveItem` | 保存/修改（含改标题） |

> **meta 标记说明**：`list` / `count` 在 meta 中标记为 `"write": true`（因 HTTP 为 POST），语义上为**查询**；`service item list` 走 `ExecuteWrite` 管线（支持 `--dry-run`），`item +list` / `item list` 走 `RunPOSTFormListMap` → `Execute`（**不支持** `--dry-run`，见 [§5.5](#55-dry-run-与-page-all-行为按代码)）。

## 4. 命令执行链路

### 4.1 示例：按标题查列表

```text
① cmd/root.go → PersistentPreRun → resolveOutputFormat（--output > cli.output > table）
② shortcuts/item/item.go → ParseBodyJSON → applyStockListDefaults
③ common.RunPOSTFormListMap("/item/stock/queryList", body)
④ internal/client → POST form {api.url}/item/stock/queryList
     Header: accessToken（auth login 写入密钥链后自动附加）
⑤ internal/output → stdout: {"ok":true,"data":...}（或 csv/ndjson 裸流）
     stderr: [debug]（--verbose）
```

### 4.2 示例：修改商品标题

```text
item save --body '{"sysItemId":123,"title":"新标题"}' [--dry-run]
  → RunPOST("/item/saveItem", body)
  → Content-Type: application/json
  → dry-run 时仅预览 URL/body（脱敏），不发送
```

### 4.3 配置与鉴权

```text
config init  → ~/.kuaimai-cli/config.yaml（api.url = erp1）
auth login   → 系统密钥链（非 config 明文）
请求头       → accessToken: <token>
```

```text
kuaimai-cli
├── config        init | get | set
```

### 5.5 dry-run 与 page-all 行为（按代码）

全局 flag：`--dry-run`、`--page-all`（`cmd/root.go` → `internal/core`）。

| 命令 | `--dry-run` | `--page-all` | 实现路径 |
|------|-------------|--------------|----------|
| `item +list` / `item list` | ❌ 仍会发请求 | ✅ | `RunPOSTFormListMap` → `Execute` |
| `item count` | ✅ 预览 form | ❌（单次 POST） | `RunPOSTFormMap` → `ExecuteWrite` |
| `item get-detail` | ❌ | ❌ | `RunGET` → `ExecuteList` |
| `item save` | ✅ 预览 JSON | ❌ | `RunPOST` → `ExecuteWrite` |
| `item update-title` | ✅ 预览 JSON | ❌ | get-detail + `ExecuteWrite` → save |
| `service item list` | ✅（meta `write: true`） | ✅（`ExecuteWrite` + form 翻页） | `servicecmd` → `ExecuteWrite` |
| `service item count` | ✅ | ❌ | `ExecuteWrite` |
| `service item get-detail` | ❌ | ❌ | `Execute` / `ExecuteList` |
| `service item save` | ✅ | ❌ | `ExecuteWrite` |
| `api POST/PUT/DELETE` | ✅ | GET 可 `--page-all` | `cmd/api` |

联调建议：**改标题等写操作**对 `item save` 使用 `--dry-run`；**列表探测**不要用 `item +list --dry-run`（无效）。复杂改写字段时仍推荐 `get-detail` → `jq` 合并后再 `save`（见 `skills/kuaimai-item/SKILL.md`）。

## 6. 输出与 Agent 约定

stdout 固定：`{ "ok", "data", "error", "hint" }`（列表 `csv`/`ndjson` 成功时为裸数据流）  
stderr：日志与友好报错（无 Go 堆栈）  
优先级：`--output` > `cli.output` > `table`

详见根目录 [AGENTS.md](../AGENTS.md) 与 `skills/kuaimai-item/SKILL.md`。

## 7. 与飞书 lark-cli 对标

### 7.1 已对齐（架构层）

kuaimai-cli 在 **CLI 骨架与 Agent 协议** 上已刻意对标飞书，差异主要在 **业务覆盖面** 与 **开箱体验**，而非「有没有 CLI 框架」。

| 能力项 | 说明 |
|--------|------|
| 分层架构 | `cmd` → `internal` → `shortcuts`，业务不进 `cmd` |
| 三级命令 | shortcuts（`+`）→ `service`（meta）→ `api`（兜底） |
| shortcuts | 手写高频 API，与浏览器 / Controller 路径一致 |
| 元数据 | `registry` + `schema` + `service` |
| 鉴权存储 | 密钥链存 `accessToken`，非 config 明文 |
| 输出协议 | stdout `{ok,data,error,hint}`；stderr 日志；无堆栈 |
| Agent 体验 | `--verbose` / `--dry-run` / `--page-all` / completion |
| Skill 平台 | `skills/` + `skill install` / `install-all` |
| 列表输出 | `csv` / `ndjson`；`item save` 等写操作 dry-run；脱敏与 `audit.log` |

> 飞书官方：[larksuite/cli](https://github.com/larksuite/cli)（README 截至 2026-05：**18** 业务域、**200+** 命令、**26** 个 Agent Skills）· [飞书 CLI 官网](https://www.feishu.cn/feishu-cli)

### 7.2 主要差异（体量与体验）

| 维度 | 飞书 lark-cli | kuaimai-cli |
|------|---------------|-------------|
| 业务域 | 18 域、200+ 命令 | 当前 **1 域**（item）、**6** 个子命令（`list` 与 `+list` 同路径；含 `update-title`） |
| Agent Skills | **26** 个（`lark-calendar`、`lark-im`…） | **2** 个（`kuaimai-shared`、`kuaimai-item`） |
| 鉴权 | OAuth 交互式 + App 凭证；`auth check` / `scopes` / `list`（多用户） | `auth login <accessToken>`；`check` / `list` / `use` / `--profile`；无 OAuth |
| Agent 登录流 | `auth login --recommend`、`--no-wait` + device code | 用户手抄 token；见 [Agent 安装指南](./kuaimai-cli-agent-installation-guide.md) |
| 身份切换 | `--as user` / `--as bot` | 未实现 |
| 后端 | 开放平台 OAPI（2500+ API） | 私有 erp-items-core @ erp1 |
| 元数据规模 | 全量 OAPI + `api` 兜底 | 内置 4 operation，可扩展 `scripts/fetch_meta` |
| 输出格式 | `--format json\|pretty\|table\|ndjson\|csv` | `--output table\|json\|csv\|ndjson`（无 `pretty`） |
| 分页 | `--page-all` + `--page-limit` + `--page-delay` | 仅 `--page-all` |
| schema 自省 | `schema <service.method>` 单方法 | 全量 `schema` |
| 实时事件 | `event consume`（NDJSON 流，Agent 子进程友好） | 未实现 |
| 工作流 Skill | 如 `lark-workflow-standup-report` | 未实现 |
| 扩展工具 | `lark-openapi-explorer`、`lark-skill-maker` | 未实现 |
| 对外 README | 根目录 `README.md` / `README.zh.md` | 根目录 [README.md](../README.md) + `docs/` 专题文档 |
| 分发 | `npx @larksuite/cli install`、npm、MIT 开源 | `npx @kuaimai-cli/cli install`、GitHub Release、[`npm/`](../npm/) |
| E2E / 升级 | `tests/cli_e2e`、升级能力 | E2E + `upgrade`/`doctor` 已有；**自动替换二进制**仍规划 |
| 安全叙事 | README 风险告知、注入防护说明 | 密钥链 + 脱敏已有；缺对外统一风险章节 |

## 10. 总结

kuaimai-cli 在 **「CLI 架构 + Agent 协议」** 上已接近飞书 lark-cli；要 **功能和体验更像飞书**，重点在于：

1. **P1**：业务域与 Skill 数量（飞书 18 域 / 26 Skill vs 当前 1 域 / 2 Skill）— 当前最大差距
2. **P0 收尾**：OAuth 式登录引导、可选 `README.zh.md`
3. **P2～P3**：pretty/分页/schema 细粒度、**自动升级二进制**
4. **P4～P5**：安全叙事、工作流 Skill、内网产品页

**不必复刻飞书 18 域**，宜采用 **「少域、深 shortcuts、强 Skill、ERP 真实场景闭环」** 策略；当前已交付 **erp-items-core 商品标题查改**（含 `update-title`、多账号、E2E），下一步建议 **第二高频业务域 + item 域深化（批量改价/上下架等）**。

理解 API 映射时：**`item *` → `shortcuts/item`；`service item *` → `meta_data.json`；`api` → 终端 path。**
